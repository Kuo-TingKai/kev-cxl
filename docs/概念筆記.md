# kev-cxl 各模組重要概念筆記

本文件整理專案中各腳本與模組的**重要概念**，便於理解與擴充。

---

## 一、QEMU 單機 CXL 模擬（qemu_cxl_single.sh）

### 1.1 CXL 與 QEMU 角色

- **CXL（Compute Express Link）**：CPU 與記憶體／儲存擴展設備之間的互連協定，支援記憶體語義（Load/Store）與池化。
- **QEMU**：在此用於**功能模擬**，模擬一台帶有 CXL Type-3 設備的 x86 主機，讓 Linux 在無實體 CXL 硬體下仍能辨識並使用 CXL 設備。

### 1.2 關鍵 QEMU 元件

| 元件 | 說明 |
|------|------|
| **pxb-cxl** | CXL 主橋（Host Bridge），接在 pcie.0 上，**僅能接 PCI/PCIe 橋接器**，不能接一般 endpoint（如 virtio-blk）。 |
| **cxl-rp** | CXL Root Port，接在 pxb-cxl 下，作為 CXL 設備的插槽。 |
| **cxl-type3** | CXL Type-3 設備（記憶體擴展／類 SSD），需綁定 **volatile-memdev**（揮發性記憶體）與 **lsa**（Label Storage Area）。 |
| **memory-backend-file** | 以檔案為後端的記憶體，供 CXL 揮發性記憶體與 LSA 使用；`share=on` 可供多 VM 共享（多主機腳本用）。 |

### 1.3 為何硬碟要接在 pcie.0

- 使用 `-drive if=virtio` 時，QEMU 會自動為 virtio-blk 選一個 PCI 插槽，可能選到 **pxb-cxl** 底下的插槽。
- pxb-cxl 只接受「橋接器」，不接受一般 endpoint，會報錯：`Only PCI/PCIe bridges can be plugged into pxb-cxl`。
- **解法**：用 `-drive if=none,id=drive0` 搭配 `-device virtio-blk-pci,drive=drive0,bus=pcie.0,addr=0x6`，明確把硬碟接在**主 PCIe 根匯流排**（pcie.0）。

### 1.4 LSA 與揮發性記憶體分離

- **LSA（Label Storage Area）**：CXL 設備上的小塊儲存區，存放設備元資料；QEMU 要求以獨立 `-object memory-backend-file,id=cxl-lsa1` 定義，並在 `-device cxl-type3` 中以 `lsa=cxl-lsa1` 引用。
- **揮發性記憶體**：模擬 CXL 的 1GB 記憶體擴展，以另一組 `memory-backend-file`（id=cxl-mem1）提供。
- 兩者必須是**不同物件**，不可只建一個後端同時給兩者用。

### 1.5 加速器：TCG 與 KVM

- **Mac（ARM）**：無法使用 KVM，使用 **TCG**（軟體轉譯），跑 x86 較慢但可驗證 CXL 邏輯。
- **Linux x86 + 具虛擬化**：腳本會自動偵測並使用 **KVM** 加速。

## 二、QEMU 多主機共享 CXL（setup_shared_mem + host_a / host_b）

### 2.1 概念

- 模擬**多台主機共用同一塊 CXL 儲存池**（對應 CXL Fabric / 資源池化）。
- 兩台 VM 透過**同一份記憶體後端檔案**（`share=on`）看到同一塊「CXL 記憶體」，可觀察共享與一致性議題。

### 2.2 共享後端

- **setup_shared_mem.sh**：在 host OS 上建立一個固定大小（預設 2G）的檔案（Linux 用 `/dev/shm`，macOS 用 `$TMPDIR`），作為兩台 QEMU 的 `memory-backend-file` 的 `mem-path`。
- **關鍵**：`-object memory-backend-file,...,share=on` 讓多個 QEMU  process 可同時映射同一檔案，形成「共享 CXL 記憶體」的模擬。

### 2.3 注意事項

- 兩台 VM 的 **bus_nr** 不同（52 與 60），避免 PCI 匯流排編號衝突。
- 真實 CXL 多主機會有一致性協定（如 CXL.cache）；此模擬僅為**共享後端**，不模擬完整 coherency，適合觀察「多主機看到同一塊記憶體」的行為。

## 三、可開機映像取得（fetch_bootable_image.sh）

### 3.1 為何需要「可開機」映像

- `qemu-img create` 只會產生**空白** qcow2，沒有開機程式與 OS，會出現 "No bootable device"。
- 腳本下載的是**已裝好 OS 的 Alpine Linux** qcow2（generic cloud / BIOS tiny），可直接從硬碟開機。

### 3.2 映像來源

- 使用 Alpine 官方 generic cloud 映像（如 dev.alpinelinux.org 的 bios-tiny），體積小、開機快。
- 此類映像為 **cloud 用途**，通常**未預設可登入的帳號／密碼**，需搭配後續的 set_alpine_root_password.sh 才能登入。

### 3.3 macOS 相容

- 若無 `/dev/shm`，腳本將映像寫入專案目錄；QEMU 與路徑無關，不影響使用。

## 四、Alpine root 密碼設定（set_alpine_root_password.sh）

### 4.1 目的

- 在**未開機**的 qcow2 上直接修改檔案系統，寫入 root 密碼，使首次開機即可用 root / 自訂密碼登入。

### 4.2 使用工具：virt-customize

- **virt-customize**（來自 libguestfs-tools）：可掛載 qcow2 並執行指令（如改密碼、寫檔），無須開機。
- 指令範例：`virt-customize -a image.qcow2 --root-password password:alpine`。

### 4.3 為何用 Docker

- macOS 上 libguestfs 安裝較麻煩，腳本改以 **Docker** 執行 Fedora 容器，在容器內安裝 libguestfs-tools 再對**掛載進去的映像**執行 virt-customize。
- 首次執行會拉取 Fedora 映像並安裝套件，約需 **5–15 分鐘**；之後若映像已存在，僅修改密碼會較快。

### 4.4 密碼自訂

- 可透過環境變數 `ALPINE_ROOT_PASSWORD` 指定密碼，例如：`ALPINE_ROOT_PASSWORD=mypass ./scripts/set_alpine_root_password.sh`。

## 五、gem5 模組（cxl_config / ruby_cxl_config / analyze_gem5）

### 5.1 gem5 的角色

- **gem5**：週期級（cycle-level）架構模擬器，用於評估**延遲、頻寬、快取一致性**等，與 QEMU 的「能跑就好」不同。
- 本專案中的腳本用於模擬 **CXL/PCIe 風格匯流排**與**多核 + Ruby 快取一致性**，供 LLM／資料庫等情境做參數實驗。

### 5.2 cxl_config.py 概念

- **SystemXBar**：模擬高頻寬匯流排（可對應 PCIe 7.0 / CXL 3.0 等級），以 `width`、`frontend_latency`、`forward_latency` 調延遲與頻寬。
- **SimpleMemory**：簡化記憶體模型（延遲 + 頻寬），可替換為 MemCtrl + DDR 等更精細模型。
- **SE 模式**：Syscall Emulation，不開完整 OS，只跑單一程式，適合做效能與協定參數測試。

### 5.3 ruby_cxl_config.py 概念

- **Ruby**：gem5 的快取一致性與記憶體子系統框架，可配置多級快取與 **MESI_Two_Level** 等協定。
- 需以 `PROTOCOL=MESI_Two_Level` 編譯 gem5 才能使用對應協定。
- 腳本為**結構範本**，實際完整 Ruby 拓撲通常需配合 gem5 內建 configs 或 Ruby 的 create_system 流程。

### 5.4 analyze_gem5.py 概念

- 解析 gem5 輸出的 **m5out/stats.txt**，以正則擷取關鍵指標：
  - **Ticks / Total_Instructions** → 可推算 CPI。
  - **L1_Misses**、**Coherence_Messages**、**Avg_Mem_Latency** → 用於評估 CXL/一致性協定對效能與流量的影響。
- 欄位名稱會隨 gem5 版本變動，若解析不到可依實際 stats 內容調整正則。

## 六、Docker 編譯環境（docker/Dockerfile）

### 6.1 目的

- 在 **Linux 容器**內提供編譯 gem5 的環境（scons、g++、Python、libboost、libhdf5 等），避免在 macOS 上直接編譯 gem5 的相容性問題。

### 6.2 使用方式

- `docker build -t gem5-dev .` 建立映像。
- 執行時以 **volume 掛載** 將本機 gem5 原始碼或腳本目錄掛進容器，在容器內執行 `scons build/X86/gem5.opt` 或執行 Python 配置腳本。
- 編譯產物可留在 volume 目錄，供 host 使用。

### 6.3 不預裝 gem5 原始碼

- Dockerfile 不預先 clone gem5，以保持映像較小；使用者自行 clone 並以 volume 掛入，在容器內編譯即可。

## 七、整體流程對照

| 階段 | 目的 | 主要模組 |
|------|------|----------|
| 取得可開機系統 | 避免 No bootable device | fetch_bootable_image.sh |
| 設定登入 | 能登入 guest 做 CXL 驗證 | set_alpine_root_password.sh |
| 單機 CXL 模擬 | 在 VM 內看到 CXL 設備並可操作 | qemu_cxl_single.sh |
| 多主機共享模擬 | 觀察多 VM 共用同一 CXL 記憶體 | setup_shared_mem.sh + host_a / host_b |
| 架構／效能評估 | 延遲、頻寬、一致性流量 | gem5 腳本 + analyze_gem5.py |
| 編譯環境 | 在 Mac 上透過容器編譯 gem5 | docker/Dockerfile |
